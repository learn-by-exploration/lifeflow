## Part 1: Expert Panel Review

### ARCHITECTS (3 reviewers)

#### Architect 1 — Structural Analysis

**Finding A1-1: God-file anti-pattern (Critical)**
`src/server.js` is a 2,410-line monolith housing schema definitions, 22 table migrations, 12 helper functions, 80+ route handlers, business logic, and the Express app setup. This is the single greatest barrier to multi-contributor work — merge conflicts are inevitable when every feature touches the same file.

**Finding A1-2: Frontend monolith (Critical)**
`public/index.html` is a 5,022-line single file containing ~900 lines of CSS, ~750 lines of HTML, and ~2,150 lines of JavaScript. Themes, component styles, DOM structure, routing, state management, 30+ render functions, and all event handlers live in one file. This is unmaintainable with >1 contributor.

**Finding A1-3: No separation of concerns (High)**
Database access, input validation, business logic, and HTTP response formatting are interleaved in every route handler. Example: the `PUT /api/tasks/:id` handler (lines 784-839) does param parsing → validation → DB query → recurring task logic → rules execution → response formatting in one block. This makes unit testing business logic impossible without HTTP.

**Finding A1-4: No module system (High)**
Zero `require()` of local modules. Everything is defined and consumed in the same scope. There is no way to import a helper function or model from another file, run it in isolation, or mock dependencies.

---

#### Architect 2 — Scalability & Maintainability

**Finding A2-1: Prepared statement leaks (Medium)**
Many route handlers call `db.prepare()` inline per request. While better-sqlite3 caches these, the code gives no indication of which queries are shared. Extracting a data-access layer would make statement reuse explicit and measurable.

**Finding A2-2: No error handling middleware (Medium)**
Each route handler catches its own errors (or doesn't). There is no Express error-handling middleware. A single `app.use((err, req, res, next) => ...)` would centralize error formatting, logging, and sanitization.

**Finding A2-3: Implicit coupling between features (Medium)**
`executeRules()` is called inline within `PUT /api/tasks/:id`. The task update handler shouldn't know about automation rules. This is a cross-cutting concern that should be event-driven or middleware-based.

**Finding A2-4: Config scattered across file (Low)**
`SETTINGS_DEFAULTS`, `GROCERY_CATEGORIES`, `LIST_TEMPLATES`, `PORT`, `dbDir` are defined at various locations (lines 7, 11, 1561, 2150, 2151). Contributors must search to find config values.

---

#### Architect 3 — API Design & Consistency

**Finding A3-1: Route ordering fragility (High)**
Express 5 matches routes top-to-bottom. Routes like `GET /api/tasks/search`, `GET /api/tasks/overdue`, `GET /api/tasks/recurring` must be defined before `GET /api/tasks/:id` or they'll be swallowed. This is documented implicitly by comments but not enforced structurally. One misplaced route in a refactor breaks the app silently.

**Finding A3-2: Inconsistent response shapes (Medium)**
- `GET /api/areas` returns enriched objects with nested goals
- `GET /api/tags` returns flat array
- `POST /api/areas` returns the created object
- `POST /api/focus` returns `{ id }`
- Some DELETE endpoints return `{ ok: true }`, others return `{ deleted: id }`
A consistent envelope (`{ data, meta, error }`) would simplify frontend error handling.

**Finding A3-3: No API versioning (Low)**
All routes are under `/api/`. No `/api/v1/` prefix. Adding breaking changes later will be painful if external consumers exist.

**Finding A3-4: Missing pagination (Medium)**
`GET /api/tasks/all`, `GET /api/activity`, and `GET /api/notes` return unbounded result sets. As data grows, these will degrade. Cursor or offset pagination should be added to list endpoints.

### TESTERS (3 reviewers)

#### Tester 1 — Test Architecture

**Finding T1-1: 40 test files with no organization (High)**
Test files are all flat in `tests/` with inconsistent naming: `batch4.test.js`, `phase2.test.js`, `exhaustive-tasks.test.js`, `tasks.test.js`. There's no directory structure, no naming convention, and overlapping coverage between files (e.g., `tasks.test.js` vs `exhaustive-tasks.test.js`).

**Finding T1-2: No test coverage measurement (High)**
There is no coverage configuration. It's impossible to know which parts of `server.js` are tested and which aren't. Node's built-in `--test-coverage` or c8 should be added.

**Finding T1-3: Test setup is shared singleton (Medium)**
`setup()` in helpers.js creates a single app+db instance. All 40 test files share it. If a test file fails to call `cleanDb()`, it poisons subsequent tests. Better-sqlite3's in-memory mode makes this fast but fragile.

**Finding T1-4: No frontend tests (Medium)**
711 tests — all server-side. There are no tests for the 2,150 lines of JavaScript in `index.html`. Render functions, state management, event handlers, and routing are untested. A lightweight DOM test (jsdom + node:test) would catch regressions.

---

#### Tester 2 — Coverage Gaps

**Finding T2-1: Automation rules engine untested end-to-end (Medium)**
`executeRules()` is 30 lines of logic (trigger matching, action execution) but has no dedicated test that creates a rule, updates a task, and verifies the rule fired.

**Finding T2-2: Search index rebuild not tested (Medium)**
`rebuildSearchIndex()` builds an FTS5 index across tasks, notes, and subtasks. No test verifies the index is correct after data changes.

**Finding T2-3: Rate limiting not tested (Low)**
`checkShareRate()` rate-limits shared list access but has no test.

**Finding T2-4: Concurrent request handling untested (Low)**
No tests verify behavior under concurrent writes (e.g., two users completing the same task simultaneously).

---

#### Tester 3 — Test Quality

**Finding T3-1: Tests lack assertion messages (Low)**
Most assertions use bare `assert.equal(a, b)` without descriptive messages. When tests fail, the output says `Expected 201, got 400` with no context about what was being tested.

**Finding T3-2: Duplicate test coverage (Medium)**
`tasks.test.js` and `exhaustive-tasks.test.js` both test task CRUD. `stats.test.js` and `exhaustive-stats.test.js` overlap. This wastes CI time and creates maintenance burden — a fix in one must be mirrored in the other.

**Finding T3-3: No negative-path tests for many endpoints (Medium)**
Many CRUD endpoints only test the happy path. Missing: creating a goal in a non-existent area, updating a task with a non-existent goal_id, deleting a tag that's linked to tasks, etc.

---

### QA (3 reviewers)

#### QA 1 — Code Quality & Standards

**Finding Q1-1: No linter or formatter (Critical for multi-contributor)**
No ESLint, Prettier, or EditorConfig. Every contributor will format differently, causing noise in diffs. This is the #1 productivity blocker for teams.

**Finding Q1-2: No CI/CD pipeline (High)**
No GitHub Actions workflow. Tests must be run manually. A `push` trigger running `npm test` is essential before accepting PRs.

**Finding Q1-3: package.json version drift (Low)**
`package.json` says `"version": "0.0.3"` but git tag is `v0.0.5`. The version should be updated.

**Finding Q1-4: .gitignore is minimal (Low)**
Only 2 entries. Missing: `*.db-wal`, `*.db-shm`, `coverage/`, `.env`, `node_modules/` (likely there but should be explicit).

---

#### QA 2 — Documentation & Onboarding

**Finding Q2-1: No CONTRIBUTING.md (High)**
New contributors have no guide for: how to run the project, how to run tests, how to structure code, where to add routes, coding style expectations.

**Finding Q2-2: No API documentation (Medium)**
80+ endpoints with no OpenAPI spec, no Postman collection, and no API reference. Contributors must read `server.js` to discover endpoints.

**Finding Q2-3: Inline comments are sparse (Low)**
The code relies on section headers (`// ─── TAGS ───`) but individual functions lack JSDoc or purpose comments. The `enrichTasks()` function (the most complex in the codebase) has no documentation.

---

#### QA 3 — DevOps & Release

**Finding Q3-1: No health metrics (Low)**
`GET /health` returns `{ status: 'ok' }` but doesn't report DB connectivity, uptime, or version.

**Finding Q3-2: Database file in repo (Medium)**
`lifeflow.db`, `lifeflow.db-shm`, `lifeflow.db-wal` are tracked by git. These should be in `.gitignore`.

**Finding Q3-3: No development/production mode distinction (Low)**
The same config runs in dev and prod. No `NODE_ENV` check, no debug logging toggle, no CORS configuration.
