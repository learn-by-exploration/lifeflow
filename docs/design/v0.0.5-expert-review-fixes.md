## Phase 1: CRITICAL — Security & Data Safety

**Files:** `src/server.js`
**New test file:** `tests/security.test.js`

### 1a. Add `express.json()` body size limit
- **Line 338:** `app.use(express.json())` → `app.use(express.json({ limit: '1mb' }))`
- Prevents payload-based DoS

### 1b. Secure `/api/import` endpoint (lines 997-1059)
- Add **confirmation token** requirement: callers must pass `{ confirm: "DESTROY_ALL_DATA" }` in body
- Import is already wrapped in a transaction (good), but add **shape validation** before the DELETE phase
- Validate that `areas`, `goals`, `tasks` are all non-empty arrays with required fields
- Refuse import if shape is invalid (return 400) — never reach the DELETE statements

### 1c. Sanitize error messages in import catch block
- **Line 1059:** `'Import failed: ' + e.message` → generic `'Import failed'`
- Never expose SQLite error details to clients

### 1d. Audit all catch blocks for schema leaks
- Search for every `res.status(500)` and `e.message` — ensure none expose SQL/table names
- Replace with generic error messages

### Tests (~12)
```
- POST /api/import without confirm token → 403
- POST /api/import with confirm:"DESTROY_ALL_DATA" + valid data → 200
- POST /api/import with malformed areas (not array) → 400
- POST /api/import with missing required fields → 400
- POST /api/import error response contains no SQL/table names
- POST /api/import is atomic: partial failure rolls back (data unchanged)
- POST with body > 1mb → 413
- POST with exactly 1mb body → 200 (not rejected)
- GET /api/import → 404 (no GET handler)
- PUT /api/import → 404 (no PUT handler)
- Verify error responses on other endpoints don't leak schema
- Import with empty tasks array → 400
```

## Phase 2: Performance

**Files:** `src/server.js`
**New test file:** `tests/performance.test.js`

### 2a. Batch habit_logs queries (lines 1327-1349)
Currently: N queries per habit per day (streak loop → up to 365 × N queries).
Fix: Batch-load all habit_logs for all habit IDs in 2 queries:
1. `SELECT habit_id, date, count FROM habit_logs WHERE habit_id IN (...) AND date >= ?` (last 400 days)
2. Build in-memory maps, compute todayCount + streak in JS

### 2b. Unify enrichTask/enrichTasks (lines 349-365 vs 366-395)
Currently: `enrichTask()` fires 4 individual queries; `enrichTasks()` batches.
Fix: Make `enrichTask(t)` call `enrichTasks([t])[0]` — one code path, always batched.

### 2c. Extract `getNextPosition()` helper
Currently: `SELECT COALESCE(MAX(position),-1)+1 as p FROM <table> WHERE <col>=?` duplicated 8+ times.
Fix: `function getNextPosition(table, scopeCol, scopeVal)` — single helper.

### Tests (~8)
```
- GET /api/habits returns correct todayCount for habit with logs
- GET /api/habits returns todayCount=0 for habit with no logs
- GET /api/habits returns correct streak (consecutive days)
- GET /api/habits returns streak=0 when yesterday not completed
- enrichTask single returns identical data to enrichTasks batch for same task
- getNextPosition returns 0 for empty table
- getNextPosition returns N+1 for populated table
- getNextPosition scoped correctly (e.g., per goal_id)
```

## Phase 3: Data Integrity

**Files:** `src/server.js`
**New test file:** `tests/data-integrity.test.js`

### 3a. Transaction: task creation + tags (lines 601-614)
Wrap `INSERT INTO tasks` + tag loop in `db.transaction()`

### 3b. Transaction: recurring task copy (lines 763-776)
Wrap new task INSERT + tag copy in `db.transaction()`

### 3c. Transaction: bulk update (lines 720-745)
Wrap the entire `for (const rawId of ids)` loop in `db.transaction()`

### 3d. Transaction: list items batch (lines 2147-2165)
Wrap the items insertion loop in `db.transaction()`

### 3e. Input validation
- **due_date**: Validate `/^\d{4}-\d{2}-\d{2}$/` format wherever accepted (task create, task update)
- **priority**: Already has CHECK constraint in schema `CHECK(priority IN (0,1,2,3))` — but add app-level validation before INSERT to give friendly 400 instead of SQLite 500
- **status**: Already has CHECK constraint `CHECK(status IN ('todo','doing','done'))` — add app-level validation
- **frequency** (habits): Validate `['daily','weekly','monthly','yearly']`
- **estimated_minutes**: Validate positive number if provided
- **target** (habits): Validate positive integer

### 3f. Circular dependency detection (lines 1152-1162)
Current: Only filters self-reference (`bid !== id`).
Fix: Before inserting deps, run DFS from each `blocked_by_id` to check if `id` is reachable → cycle detected → return 400.

### Tests (~15)
```
- Task+tag creation: if tag insert fails (bad tag_id), task is rolled back
- Recurring copy: tags are always copied atomically with new task
- Bulk update: partial ID failure rolls back all changes
- List items batch: partial failure rolls back entire batch
- Task with invalid due_date format → 400
- Task with valid due_date format → 201
- Task with priority=5 → 400
- Task with priority=2 → 201
- Task with status='invalid' → 400 (on PUT update)
- Habit with frequency='never' → 400
- Habit with target=0 → 400
- Task with estimated_minutes=-5 → 400
- Circular dep: A→B→C→A → 400
- Self-reference dep: A→A → filtered out
- Valid dep chain A→B→C → succeeds
```
