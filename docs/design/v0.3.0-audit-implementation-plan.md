## Phase 0 — Shared Utilities

**Must be built first — unblocks all sprints.**

### `withSubmit(btnId, fn)` — universal form submit wrapper

```js
async function withSubmit(btnId, fn) {
  const btn = $(btnId);
  if (!btn || btn.disabled) return;
  const orig = btn.innerHTML;
  btn.disabled = true;
  btn.innerHTML = '<span class="btn-spinner"></span><span class="btn-label">Saving…</span>';
  try { await fn(); }
  catch(e) { showToast(e?.message || 'Something went wrong — please try again'); throw e; }
  finally { btn.disabled = false; btn.innerHTML = orig; }
}
```

Add after `debounce` function in `public/app.js`.

### `loggedCatch(label, userVisible)` — replaces silent `.catch(()=>{})`

```js
function loggedCatch(label, userVisible = false) {
  return (e) => {
    console.warn('[LifeFlow]', label, e);
    if (userVisible) showToast('Failed to load ' + label);
  };
}
```

### CSS spinner (add to `public/styles.css`)

## Sprint 2 — Backend Integrity

**Goal:** Pagination on unbounded endpoints + validation gaps closed.
**Estimated effort:** 1 day
**Parallel with Sprint 1** — no frontend dependencies.

### Pagination strategy

**Backward-compatible** — envelope only returned when `?limit=` is explicitly provided. Existing callers get flat arrays unchanged.

```
GET /api/tasks/all              → []  (unchanged, all existing callers safe)
GET /api/tasks/all?limit=50     → { items: [], total: N, hasMore: bool, offset: 0 }
```

| Endpoint | File | Current | Fix |
|----------|------|---------|-----|
| `GET /api/tasks/all` | `src/routes/tasks.js:20` | Returns all rows, no limit | Add optional `?limit` + `?offset` |
| `GET /api/tasks/recurring` | `src/routes/tasks.js:132` | Returns all rows | Same pattern |
| `GET /api/goals` | `src/routes/areas.js:133` | Returns all active goals | Same pattern (cap default at 200 for dropdown) |

### Validation gaps (`src/routes/productivity.js`)

| Route | Missing | Fix |
|-------|---------|-----|
| `POST /api/rules` | `name` not trimmed; no allowlist on `trigger_type`/`action_type` | Trim + max 100 chars; whitelist against constant array |
| `PUT /api/rules/:id` | No integer ID check; same field gaps | Integer check + same validation as POST |
| `PUT /api/notes/:id` | Empty string `title` passes through | Guard `if (!title.trim()) return 400` |
| `DELETE /api/notes/:id` | No integer ID check (NaN passes silently to SQLite) | `Number.isInteger(id)` check |
| `DELETE /api/inbox/:id` | Same as above | Same fix |
| `PUT /api/inbox/:id` | `priority` accepts any value | Range check `0–3` |

Define allowlists as constants at top of `productivity.js`:
```js
const VALID_TRIGGER_TYPES = ['task_completed','task_created','task_overdue','task_updated'];
const VALID_ACTION_TYPES  = ['add_tag','set_priority','move_to_goal','send_notification'];
```

### Acceptance criteria
- `/api/tasks/all` with `?limit=50` returns `{items, total, hasMore}` shape
- All existing tests pass unchanged (no breaking API change)
- Whitespace-only rule name returns 400
- Non-integer IDs on delete routes return 400, not silent success

## What to Defer

These are valid improvements that should not block the 5 sprints:

- **Virtual scrolling** for 500+ task lists — Sprint 2 pagination unlocks the design decision first
- **Focus trap in modals** — medium a11y effort, low user impact for single-user local app
- **Undo toast for subtask delete** — `confirm()` in Sprint 1 is sufficient; deferred async delete is a separate complexity
- **`<details>/<summary>` for detail panel** — CSS class approach in Sprint 5 is simpler and consistent with existing patterns
- **`/api/tasks/board` rate limits** — already debounced in Sprint 1; per-endpoint rate limiting is overkill for personal use
